WCAG 2.1 Reference

The web works best when it works for everyone.

This guide covers the main WCAG guidelines, what tends to go wrong on real sites, and some interactive tools to actually feel what inaccessible design is like from a user's perspective.

01 — Guidelines

What the guidelines actually cover

WCAG 2.1 is organized around four principles — Perceivable, Operable, Understandable, and Robust (POUR). Everything falls under one of them. Here's what that looks like on real websites.

Text is the backbone of most websites, and it's also one of the most commonly broken things from an accessibility standpoint. The issues usually aren't dramatic — they're small decisions that stack up.

  • Minimum font size: 16px for body text is a solid baseline. WCAG doesn't name a specific size, but anything under 12px is almost always a problem.
  • Line height: At least 1.5x the font size for body text. Tight line spacing makes reading significantly harder for people with dyslexia.
  • Line length: Keep paragraphs between 60 and 80 characters wide. Very long lines force the eye to travel too far, which causes reading errors.
  • Text spacing: Users need to be able to override letter spacing, word spacing, and line height without content breaking.
  • Avoid justified text: Full justification creates irregular gaps between words that disrupt reading flow.
  • Don't use text inside images: Screen readers can't read it, it doesn't scale, and it breaks at high zoom levels.
Common mistake

Setting user-select: none on body text. Users who rely on text-to-speech extensions need to be able to select and copy text.

Quick fix

Add this to your CSS: p { max-width: 70ch; line-height: 1.6; }. That one rule handles line length and spacing for most body text.

Screen reader users often navigate entirely by jumping between headings — think of it like skimming. If your heading structure is broken, they're basically flying blind.

  • One H1 per page: The H1 is the page title. Using more than one confuses both screen readers and search engines.
  • Don't skip levels: H1 → H2 → H3 in order. Jumping from H2 to H4 breaks the outline.
  • Headings are for structure, not style: Don't choose heading levels based on how big you want text to look. That's what CSS is for.
  • Every section needs a heading: If there's a long section with no heading, screen reader users have no way to identify or navigate to it.
<!-- Correct -->
<h1>Accessibility Guide</h1>
  <h2>Text Guidelines</h2>
    <h3>Font Size</h3>
  <h2>Color & Contrast</h2>

<!-- Wrong -->
<h1>Accessibility Guide</h1>
  <h3>Text Guidelines</h3> <!-- skipped H2 -->
Quick check

Open dev tools, search for all heading tags, and read them in order out loud. If they don't form a logical outline of the page, the structure needs fixing.

Contrast is the ratio between text brightness and its background. Low contrast is one of the most common WCAG failures — and honestly, it affects everyone when you're on your phone in sunlight, not just people with visual impairments.

  • Normal text (under 18pt): Minimum 4.5:1 contrast ratio.
  • Large text (18pt+ or 14pt+ bold): Minimum 3:1 contrast ratio.
  • UI components and icons: Minimum 3:1 against adjacent colors.
  • Never use color alone to convey meaning: "Required fields are shown in red" fails colorblind users. Add an asterisk or label too.
  • Focus indicators need contrast too: The outline around a focused button must meet 3:1 against the surrounding area.
Very common failure

Placeholder text in form inputs. It's almost always low-opacity gray that fails contrast requirements. Use real visible labels — placeholders aren't a substitute anyway.

When alt text is missing, screen readers often fall back to reading the filename — something like "IMG_20240311_142203.jpg". Not exactly useful.

  • Informative images: Describe what the image conveys, not just what it looks like. For a chart, describe the trend.
  • Decorative images: Use alt="" so screen readers skip it. Don't leave alt out entirely — that's different.
  • Functional images: Alt text should describe the destination or action. A logo linking to the homepage should say "Home", not "Company logo".
  • Complex images: Charts and diagrams need a longer text description nearby or via aria-describedby.
<!-- Informative -->
<img src="chart.png"
     alt="Bar chart: revenue up 40% in Q3">

<!-- Decorative -->
<img src="divider.png" alt="" role="presentation">

<!-- Linked logo -->
<a href="/"><img src="logo.png" alt="Home"></a>

Many people can't use a mouse — motor impairments, tremors, or just preference. Every interactive element needs to be reachable and usable with a keyboard alone.

  • Tab order must make sense: Tab should move through interactive elements in a logical order — typically left to right, top to bottom.
  • Never remove focus outlines: outline: none without a custom replacement is genuinely harmful. Keyboard users need to see where they are.
  • Custom components need keyboard support: Dropdowns, modals, sliders — all need keyboard event handlers. Enter and Space activate buttons. Arrow keys move through listboxes.
  • Modals must trap focus: When a modal opens, focus should stay inside it. Users shouldn't be able to Tab to elements behind the overlay.
  • Skip links: A "Skip to main content" link at the very top lets keyboard users bypass nav menus every time they load a page.
Test it yourself

Put your mouse away and try navigating your site with only Tab, Enter, Space, and arrow keys. Most developers find at least two or three broken interactions the first time they try this.

Forms are where a lot of accessibility failures pile up because there are so many small things to get right. Filling out an inaccessible form with a screen reader is one of the most frustrating experiences on the web.

  • Every input needs a label: Placeholder text is not a label. Once a user starts typing, the placeholder disappears and there's no way to check what the field was asking for.
  • Associate labels programmatically: Use <label for="inputId"> or wrap the input inside the label. aria-label is a fallback, not a first choice.
  • Error messages must be specific: "Invalid input" is useless. "Email must include an @ symbol" tells the user exactly what to fix.
  • Required fields: Mark them with required or aria-required="true". Explain what asterisks mean at the top of the form.
  • Group related inputs: Use <fieldset> and <legend> for radio buttons and checkboxes. Screen readers announce the legend before each option.
Common failure

Auto-focusing a field on page load can disorient screen reader users who haven't had a chance to understand where they are yet. Avoid it unless the whole point of the page is that specific form.
