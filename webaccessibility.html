WCAG 2.1 Reference

The web works best when it works for everyone.

Web accessibility is about making sure people with disabilities can use the web just as well as everyone else. That covers visual impairments, hearing loss, motor difficulties, and cognitive differences. This guide breaks down the main guidelines, explains what actually goes wrong on most sites, and gives you practical ways to fix it.

01 — Guidelines

What the guidelines actually cover

WCAG 2.1 is organized around four principles. Everything falls under one of them: content must be Perceivable, Operable, Understandable, and Robust (POUR). Here's what that means for real websites.

Text is the backbone of most websites, and it's also one of the most commonly broken things from an accessibility standpoint. The issues usually aren't dramatic — they're small decisions that stack up.

  • Minimum font size: 16px for body text is a solid baseline. WCAG doesn't name a specific size, but anything under 12px is almost always a problem.
  • Line height: At least 1.5x the font size for body text. Tight line spacing makes reading significantly harder for people with dyslexia or cognitive disabilities.
  • Line length: Keep paragraphs between 60 and 80 characters wide. Very long lines force the eye to travel too far, which causes reading errors.
  • Text spacing: Users need to be able to override letter spacing, word spacing, and line height without content breaking or overlapping.
  • Avoid justified text: Full justification creates irregular gaps between words that disrupt reading flow for people with dyslexia.
  • Don't use text inside images: Screen readers can't read it, it doesn't scale, and it breaks at high zoom levels.
Common mistake

Setting user-select: none or pointer-events: none on body text. Users who rely on text-to-speech browser extensions need to be able to select and copy text.

Quick fix

Add this to your CSS: p { max-width: 70ch; line-height: 1.6; }. That single rule handles line length and spacing for most body text.

Screen reader users often navigate pages entirely by jumping between headings, similar to how sighted users skim. If your heading structure is wrong, these users get a completely broken experience.

  • One H1 per page: The H1 is the page title. Using more than one confuses both screen readers and search engines.
  • Don't skip levels: H1 should be followed by H2, H2 by H3, and so on. Jumping from H2 to H4 breaks the outline structure.
  • Headings are for structure, not style: Don't pick heading levels based on how big you want text to look. Style with CSS, use headings for hierarchy.
  • Every section needs a heading: If a user is navigating by heading and there's a long section with no heading, they have no way to identify it.
<!-- Correct -->
<h1>Accessibility Guide</h1>
  <h2>Text Guidelines</h2>
    <h3>Font Size</h3>
  <h2>Color & Contrast</h2>

<!-- Wrong -->
<h1>Accessibility Guide</h1>
  <h3>Text Guidelines</h3> <!-- skipped H2 -->
Quick check

Open your browser's developer tools and search the page for all heading tags. Read them in order out loud. If they don't form a logical outline of the page, fix the structure.

Contrast is the ratio between the brightness of foreground text and its background. Low contrast is one of the most common WCAG failures, and it affects everyone in difficult lighting conditions, not just people with visual impairments.

  • Normal text (under 18pt): Minimum 4.5:1 contrast ratio against the background.
  • Large text (18pt+ or 14pt+ bold): Minimum 3:1 contrast ratio.
  • UI components and icons: Minimum 3:1 against adjacent colors.
  • Never use color alone to convey meaning: "Required fields are shown in red" fails users who are colorblind. Add an asterisk or label too.
  • Focus indicators need contrast too: The outline around a focused button must meet the 3:1 requirement against the surrounding area.
Very common failure

Placeholder text in form inputs. Placeholder text is almost always styled with low-opacity gray that fails contrast requirements. Use visible labels instead, or style placeholders to meet the 4.5:1 threshold.

Screen readers announce images by reading their alt text. If alt text is missing, the reader often reads out the filename instead — something like "IMG_20240311_142203.jpg", which is useless. If it's an SVG or icon with no alt, it might just say "image".

  • Informative images: Write alt text that describes what the image conveys in context, not just what it looks like. For a chart, describe the trend, not the colors.
  • Decorative images: Use an empty alt attribute — alt="" — so screen readers skip it. Don't just leave alt out entirely.
  • Functional images (linked or clickable): The alt text should describe the destination or action, not the image itself. A logo that links to the homepage should say "Home", not "Company logo".
  • Complex images: Charts, diagrams, and infographics need a longer text description nearby — either visible on the page or via aria-describedby.
<!-- Informative image -->
<img src="chart.png"
     alt="Bar chart showing revenue growth of 40% in Q3">

<!-- Decorative image -->
<img src="divider.png" alt="" role="presentation">

<!-- Linked logo -->
<a href="/"><img src="logo.png" alt="Home"></a>

When a screen reader reads a table, it announces each cell along with its row and column headers so the user knows what the data means. Without proper markup, users just hear a stream of numbers or values with no context.

  • Use <th> for headers: Don't style a regular <td> to look like a header. The <th> element gives it semantic meaning that assistive tech can use.
  • Add scope attributes: scope="col" on column headers and scope="row" on row headers tells screen readers how headers relate to cells.
  • Add a <caption>: This is the table's title and the first thing a screen reader announces. Without it, users don't know what the table is about before diving in.
  • Never use tables for layout: CSS grid and flexbox exist for layout. Using tables for layout confuses screen readers that expect table structure to mean data relationships.
<table>
  <caption>Q3 Sales by Region</caption>
  <thead>
    <tr>
      <th scope="col">Region</th>
      <th scope="col">Revenue</th>
    </tr>
  </thead>
  <tbody>
    <tr>
      <th scope="row">Northeast</th>
      <td>$420,000</td>
    </tr>
  </tbody>
</table>

A screen reader converts on-screen content into speech or braille output. It reads the accessibility tree — a version of the DOM that contains only semantic information. Visual styling is mostly invisible to it.

  • Announce dynamic changes: If content updates without a page reload (like a form error appearing), use aria-live="polite" so screen readers announce the change.
  • Hidden content: display: none and visibility: hidden hide content from screen readers too. To hide visually but keep accessible, use the "visually hidden" CSS class instead.
  • Icon-only buttons: A button with only an icon inside it will be announced as "button" with no description. Always add aria-label="Close menu" or similar.
  • Reading order: Screen readers follow the DOM order, not the visual order. If you reorder elements visually with CSS, the reading order can become completely nonsensical.
  • ARIA roles: Use ARIA roles to add semantic meaning when HTML elements alone aren't enough — but always prefer native HTML elements first. A <button> is better than a <div role="button">.
/* Keep content accessible but visually hidden */
.visually-hidden {
  position: absolute;
  width: 1px;
  height: 1px;
  padding: 0;
  margin: -1px;
  overflow: hidden;
  clip: rect(0,0,0,0);
  border: 0;
}

<!-- Announce dynamic changes -->
<div aria-live="polite" aria-atomic="true">
  Form submitted successfully.
</div>

Many people can't use a mouse — whether because of motor impairments, tremors, or simply preference. Every interactive element on a page needs to be reachable and operable with a keyboard alone.

  • Tab order must make sense: Pressing Tab should move through interactive elements in a logical order — usually left to right, top to bottom.
  • Never remove focus outlines entirely: outline: none without a custom replacement is one of the most harmful things you can do. Keyboard users need to see where they are.
  • Custom components need keyboard support: A dropdown menu, modal, or custom slider built in JavaScript needs keyboard event handlers. The Enter and Space keys should activate buttons. Arrow keys should move through options in a listbox.
  • Modals must trap focus: When a modal opens, focus should stay inside it. The user should not be able to Tab to elements behind the overlay.
  • Skip links: A "Skip to main content" link at the very top of the page lets keyboard users bypass navigation menus on every load. It can be visually hidden until focused.
Test it yourself

Close your mouse right now and try to use your own website with only the Tab, Enter, Space, and arrow keys. Most developers find at least two or three broken interactions the first time they try this.

Forms are where a lot of accessibility failures happen because there are so many small things to get right. A user relying on a screen reader filling out an inaccessible form is one of the most frustrating experiences on the web.

  • Every input needs a label: Placeholder text is not a label. When a user starts typing, the placeholder disappears and there's no way to check what the field was asking for.
  • Associate labels programmatically: Use <label for="inputId"> or wrap the input inside the label element. aria-label is a backup, not the first choice.
  • Error messages must be specific: "Invalid input" tells the user nothing. "Email address must include an @ symbol" tells them exactly what to fix.
  • Required fields: Mark them with required or aria-required="true". If you use an asterisk (*), explain it at the top of the form.
  • Group related inputs: Use <fieldset> and <legend> to group related radio buttons or checkboxes. Screen readers announce the legend before each option.
Common failure

Auto-focusing on a field when the page loads can disorient screen reader users who haven't had a chance to understand the page context yet. Avoid it unless the primary purpose of the page is that specific form.
